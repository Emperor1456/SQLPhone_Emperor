## 🧱 BRICK 1 – Single‑Row Insert & Explicit Column Lists

`INSERT INTO` adds new rows to a table.  
Always specify the column list — it protects your statement if the table structure changes.

**① Create the cities table**
```sql
CREATE TABLE cities (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    country TEXT,
    population INTEGER
);
```
- `id` auto‑assigns when you insert NULL.
- `population` can be NULL — not every city’s census is complete.

**② Insert three major cities in one batch (Easy practice)**
```sql
INSERT INTO cities (name, country, population)
VALUES
    ('Tokyo', 'Japan', 13929286),
    ('Delhi', 'India', 16787941),
    ('Shanghai', 'China', 24183300);
```
This single statement inserts three complete rows.  
The column list `(name, country, population)` matches the order of the values.  
`id` is omitted, so SQLite assigns 1, 2, and 3 automatically.

Verify:
```sql
SELECT * FROM cities;
```
You’ll see three rows, each with a generated `id`.

**③ Add a city with unknown population (Medium practice)**
```sql
INSERT INTO cities (name, country)
VALUES ('London', 'UK');
```
- Only `name` and `country` are provided.
- `population` defaults to NULL — the empire awaits the census report.

Check the result:
```sql
SELECT * FROM cities;
```
London appears with a NULL population. This is expected and valid; our schema allows it.

> 💡 **INSIGHT:** Multi‑row inserts are faster and cleaner than three separate `INSERT` statements. Use them whenever you add multiple related rows.

> ⚠️ **WARNING:** If you omit the column list entirely (`INSERT INTO cities VALUES (...)`), you must provide values for **all** columns in the exact order they were created. This is fragile — always list columns explicitly.

## 🧱 BRICK 2 – Returning Auto‑Generated IDs

When a primary key is generated by SQLite, you often need to know what ID was assigned. Use the `RETURNING` clause (SQLite 3.35+).

**④ Insert a fifth city and capture its ID (Hard practice)**
```sql
INSERT INTO cities (name, country, population)
VALUES ('Paris', 'France', 2161000)
RETURNING id;
```
The database inserts the row and immediately returns the assigned `id` (likely 5).  
No separate `SELECT` needed — one round trip, one result.

**⑤ Use the returned ID in application code**  
In Python, you’d execute the query and fetch the result:
```python
cursor.execute("INSERT INTO cities (...) VALUES (...) RETURNING id")
new_id = cursor.fetchone()[0]
```
This pattern is essential for creating dependent rows (e.g., inserting an order, then using `order_id` to insert line items).

**⑥ Insert from another table**  
You can also copy data between tables:
```sql
INSERT INTO cities (name, country, population)
SELECT city_name, nation, residents FROM temp_cities;
```

> 💡 **ADVANCED TIP – Bulk inserts with scripts:**  
> Put all your `INSERT` statements in a `.sql` file and run it with `sqlite3 geography.db < seed.sql`. This is how you’ll seed Companion’s memory with initial knowledge.

> ⚠️ **WARNING:** `RETURNING` is available in SQLite 3.35.0 and later. Termux’s `sqlite` package is usually up‑to‑date, but verify with `sqlite3 --version` if you get a syntax error.

## 💡 Real‑world Usage

**Banking – add new accounts with auto‑ID**
```sql
INSERT INTO accounts (owner, balance)
VALUES ('Emperor', 5000)
RETURNING account_id;
```

**E‑commerce – bulk load product catalog**
```sql
INSERT INTO products (sku, name, price)
VALUES ('SKU-1','Widget',9.99),
       ('SKU-2','Gadget',19.99),
       ('SKU-3','Sprocket',14.50);
```

**Logistics – record new shipments**
```sql
INSERT INTO shipments (destination, weight)
VALUES ('Dhaka', 12.5)
RETURNING tracking_id;
```

**HR – onboard multiple employees at once**
```sql
INSERT INTO employees (name, department, salary)
VALUES ('Rahim','Engineering',60000),
       ('Karim','Logistics',45000);
```

## 📌 Key Takeaway
- `INSERT INTO table (columns) VALUES (...)` adds rows.
- Always specify the column list — it’s safer and more readable.
- Multi‑row inserts are faster and cleaner than multiple single inserts.
- `RETURNING` retrieves generated IDs without an extra query.
